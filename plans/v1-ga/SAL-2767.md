## Acceptance criteria
- Dispatch HTTP client wraps ollama calls with tenacity retry on 5xx (3 retries, exponential backoff with jitter)
- 4xx errors bypass retry (surface fast)
- Each retry emits `[infra_retry]` log with target URL + attempt number + model
- Final 3-retry exhaustion: original `httpx.HTTPStatusError` re-raised so existing fallback chain in `call` / `call_with_tools` engages on a different model
- Test: mock 500-500-200 succeeds at attempt 3 on the SAME model (no fallback)
- Test: mock 400 fails immediately with no retry (1 request total)
- Test: mock 500-500-500 exhausts retry, fallback layer engages on a different model
- Test: `[infra_retry]` log line contains URL + attempt counter
- Test: `_is_retryable_infra_error` predicate locks the 4xx-vs-5xx + connection-class boundary

## Risks
- **Latency under sustained 5xx:** worst case adds ~3.5s (0.5 + 1 + 2 with jitter) before the fallback engages. Mitigation: `_INFRA_RETRY_MAX_SECONDS=4.0` caps the per-attempt wait; total wall-clock bounded.
- **Cost under 5xx storms:** each retry is a full request to the gateway; if the gateway is hard-down, we burn 3x the requests before falling back. Mitigation: connection-class errors (refused, EOF) retry too, so a hard-down gateway short-circuits fast on TCP.
- **Tenacity coupling:** wrapper uses `tenacity.AsyncRetrying`; tenacity is already a pinned dep (`pyproject.toml`), so no new dependency surface.
- **Out of scope for this PR:** the Linear ticket also calls for an `infra_exhausted` exception class distinct from `model_failed` plus a `main.py` mapping to respawn-once. That is deferred to a follow-up so this PR stays additive (~30 LOC code) and doesn't change the existing fallback semantics. The retry wrapper here is the foundation; the respawn classification rides on top.
