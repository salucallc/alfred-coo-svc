# SAL-3075: Blackout recovery monotonicity

## Target paths
- tests/fleet/e2e/test_multitenant.py
- plans/v1-ga/C-29b.md

## Acceptance criteria
1. `tests/fleet/e2e/test_multitenant.py` contains `test_multitenant_blackout_recovery` (or `test_multitenant_blackout_*`).
2. `pytest tests/fleet/e2e/test_multitenant.py -v -k "blackout"` exits 0.
3. After test run: `jq '.tenant_id' e2e_audit.jsonl | sort -u | wc -l` returns exactly `2`.
4. After test run: per-tenant `local_seq` rows from `fleet_memory_sync_log` are gap-free (asserted in test).
5. CI step records F21 suite wall-clock and fails on over 900s; the recorded value is captured in the CI log.
6. JUnit XML contains the `fleet_multitenant_blackout` test case label.

## Verification approach
- Run the extended E2E test suite with `pytest -k blackout` and ensure it passes.
- Verify audit JSON tenant count via `jq` as part of the test.
- Query the DB for per-tenant `local_seq` monotonicity inside the test.
- Add a CI timing wrapper to enforce < 900 seconds total suite duration.

## Risks
- Increased test runtime may approach CI timeout; mitigated by `E2E_FAST=1` fast mode.
- Placeholder implementation does not exercise real blackout; may need future fleshing.
