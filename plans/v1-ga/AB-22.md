# AB-22: Consumer Key Minting & Verification

## Target paths
- src/alfred_coo/__init__.py
- plans/v1-ga/AB-22.md (new)

## Acceptance criteria
- [ ] First boot mints all four consumer keys; second boot is a no‑op.
- [ ] File permissions verified 0600 on POSIX; skipped on Windows with warn.
- [ ] `verify()` returns `None` for unknown tokens; returns populated `ConsumerIdentity` for known.
- [ ] Unit tests cover: mint, reload, verify‑hit, verify‑miss, idempotent re‑boot.
- [ ] Secret is not stored on disk (only hash baked into key string).

## Verification approach
- Run the module import and confirm that ``mint_all`` populates four keys exactly once.
- Call ``reload_keys`` and ensure no additional keys are generated.
- Use ``verify`` with a known key (from ``mint_all``) and an unknown token, asserting the correct return types.
- Execute ``check_file_permissions`` on a temporary file with mode 0o600 and on Windows to observe the warning path.
- Inspect the source to verify no secret write operations to the filesystem.

## Risks
- In‑memory only storage means keys disappear on process restart; this is intentional for the GA ticket.
- Permission check may behave unexpectedly on non‑POSIX platforms; warning ensures visibility.
