# SAL-2842: [AB-17-y] Reconcile orphan-active tickets with no child_task_id (post-v7l)

Linear: https://linear.app/saluca/issue/SAL-2842
Branch: `feature/poll-children-completion-no-pr-url`
Related: SAL-2831 / PR #81 (deficient — child_task_id-required), SAL-2832 (prompt-drift root cause)

## Bug — third occurrence on 2026-04-25

**v7l** (kickoff `cf1dd345-99c7-4455-80a6-1f9974ea426b`, started 15:18 UTC). Wave 1 dispatched 1 ticket (SS-11), which respawned to fix-round-1 at 15:39 UTC, completed without PR URL at 15:40:59, and was correctly transitioned to FAILED by `_poll_children` at 15:41:15 (per `ticket_failed: identifier=SAL-2672, note='child completed without PR URL'` in soul state events at ts 1777131675.13). The existing complete-with-no-PR path **worked correctly**.

But the watchdog kept reporting `[watchdog] in_flight=1 ready=0` continuously from 15:56:49 onwards (70+ minutes and counting at PR open time). Investigation of the persisted state revealed the actual stuck ticket: not SS-11 but **SAL-2603 (ALT-06 GitHub PR poller, UUID `28b30b6e-836a-471b-961a-2ccbec9160ff`)**.

SAL-2603 hydrated as `in_progress` from a prior daemon's persisted state (every checkpoint since the first one at 15:18:36 shows it as `in_progress`). Across all 91 soul checkpoints emitted under `autonomous_build:cf1dd345-...:state`, **`dispatched_child_tasks` never contained an entry for SAL-2603's UUID**. Yet `ticket_status[28b30b6e]` remained `in_progress` indefinitely.

## Root cause — definitively isolated

`_poll_children` filters its in-flight set at `orchestrator.py:2003-2007`:

```python
in_flight = [
    t for t in self.graph.nodes.values()
    if t.child_task_id
    and t.status not in TERMINAL_STATES
]
```

A ticket with `child_task_id is None` AND status in {DISPATCHED, IN_PROGRESS, PR_OPEN, REVIEWING, MERGE_REQUESTED} is **silently excluded** from this set. The AB-17-x phantom-child reconciler (PR #81) lives inside the same loop, so its `if rec is None: ...phantom force-fail` branch is also unreachable for orphan-active tickets.

`_in_flight_for_wave` (`orchestrator.py:1768-1776`), used by the watchdog and dispatch-cap accounting, filters on **status alone** with no child_task_id check. So the orphan ticket counts as in-flight forever.

The deadlock-break at `orchestrator.py:1678-1707` (AB-17-n) requires `not in_flight and not ready`; with `in_flight=1` permanently, this exit path is also blocked.

Net effect: a ticket can remain in an active state with no child task and no recovery mechanism for the lifetime of the orchestrator process.

How does a ticket end up active with no child_task_id? At least three vectors are possible:

1. **State-restore inheritance.** A ticket persisted as `in_progress` in a prior daemon's checkpoint was hydrated by `_apply_restored_status` (line 1486-1490). Hydration only sets `node.child_task_id = child_id` when `state.dispatched_child_tasks[uuid]` is truthy; if the prior daemon snapshot had no entry there (e.g. the prior process died between status flip and the next `_snapshot_graph_into_state` call), the restored ticket has status=IN_PROGRESS with `child_task_id=None`. This is the v7l case.
2. **Mid-flight crash.** `_dispatch_child` sets `ticket.status = TicketStatus.DISPATCHED` then `ticket.child_task_id = resp["id"]`. A crash between those two lines, before checkpoint, would persist DISPATCHED with no child id.
3. **External Linear edit.** If an operator manually moves a Linear ticket through statuses while the daemon is offline, and a future `_build_graph` run picks up the in-progress state, the ticket can land in IN_PROGRESS with no child id from the orchestrator's perspective.

Regardless of vector, the recovery class is the same.

## Fix scope

Add `_reconcile_orphan_active()` as a **pre-pass** to `_poll_children` (runs BEFORE the existing in_flight filter). Force-fails any ticket in `ACTIVE_TICKET_STATES` (new module-level constant, mirrors `_in_flight_for_wave`'s status set) whose `child_task_id is None` AND whose `_ticket_transition_ts` is older than `STUCK_CHILD_FORCE_FAIL_SEC` (30 min, same window as AB-17-x).

Sub-threshold orphans are intentionally tolerated: the dispatch loop may legitimately transition a ticket to DISPATCHED before the next snapshot stamps `child_task_id` into state, so the no-child window has a one-tick race that should not produce false failures.

The `if not in_flight: return []` short-circuit was widened to `return list(orphan_failed)` so the orphan reconciler still surfaces its updates when the AB-17-x in-flight set is empty (which is the v7l shape: one orphan, zero in-flight with child ids).

The orphan-failed list is rolled into the existing `updated` return so the watchdog progress timestamp bumps on the same tick.

## Linear roll-back

Force-failed orphan tickets get `_update_linear_state(ticket, "Backlog")` so the Linear issue returns to the PM/operator queue for diagnosis (matching the AB-17-x phantom-child handling).

## What was ruled out

- **SS-11 / SAL-2672 stuck on missing PR URL**: the previous sub's hypothesis. Refuted by soul checkpoint events showing the `child completed without PR URL` path fired correctly at 15:41:15.
- **soul-svc read-after-write lag**: irrelevant for the orphan case (no mesh lookup happens for orphan-active tickets).
- **AB-17-n deadlock break**: gated on `not in_flight`, blocked by the phantom in-flight count.
- **AB-17-q external cancel**: Cristian's standing instruction is "do not restart"; the cancel-exit also requires in_flight to drain, which it cannot.
