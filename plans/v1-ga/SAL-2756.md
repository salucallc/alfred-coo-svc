# SAL-2756: [AB-17-q] External cancel signal for orchestrator coroutine

## Target paths

- `src/alfred_coo/mesh.py` — add `MeshClient.get_task(task_id)` helper
- `src/alfred_coo/autonomous_build/orchestrator.py` — cancel polling +
  graceful-drain exit + `_complete_kickoff_canceled` terminal handler
- `tests/test_autonomous_build_orchestrator.py` — fake-mesh-driven test
  asserting drain semantics

## Acceptance criteria

- An external cancel signal (one of: kickoff status `canceled`, status
  `failed` with `result.cancel == true`, or just status `failed`) is
  observed by the orchestrator within one `poll_sleep_sec` tick (default
  45s, ~30s in fast configs) without a daemon restart.
- Orchestrator stops dispatching new wave members from the moment cancel
  is observed (`_drain_mode` flips on).
- In-flight children (already dispatched mesh tasks) are allowed to
  complete normally — cancel does NOT call `task.cancel()` on the
  asyncio Task, so the existing soul-svc lifecycle for child tasks is
  preserved.
- Once in-flight drains, `_dispatch_wave` exits the loop without
  raising; the wave gate is skipped (it would raise on
  never-dispatched tickets); `_run_inner` calls
  `_complete_kickoff_canceled` instead of `_run_on_all_green_actions` +
  `_complete_kickoff`.
- The kickoff record ends in `status=failed` with `result.cancel=true`
  + `result.cancel_reason` populated and `final_state_snapshot`
  attached for post-mortem.

## Verification approach

1. Unit-test (`test_orchestrator_external_cancel_drains_and_exits`)
   drives `_dispatch_wave` with a `_FakeMesh` that returns a `failed`
   record for the kickoff after N polls. Asserts:
   - new dispatches stop after the cancel tick (no further entries in
     `mesh.created` for tickets not yet dispatched);
   - in-flight children that were dispatched before cancel complete
     normally (their tickets reach terminal states);
   - `state.events` contains `cancel_requested` + `wave_dispatch_canceled`;
   - `_complete_kickoff_canceled` posts `mesh.complete(status="failed",
     result={"cancel": true, ...})` once.
2. Live verification (operator workflow):
   - Kick off an autonomous-build run (e.g. `v8-smoke`) and observe a
     wave dispatching children.
   - From another shell:
     ```
     curl -X PATCH http://100.105.27.63:8080/v1/mesh/tasks/<KICKOFF_ID>/complete \
       -H "Authorization: Bearer $SOUL_API_KEY" \
       -H "Content-Type: application/json" \
       -d '{"session_id":"alfred-main","status":"failed","result":{"cancel":true,"reason":"manual_test"}}'
     ```
   - Within 45s the daemon log emits `[cancel] external cancel signal
     observed for kickoff ...` and `[cancel] wave N drained (no
     in-flight)`. The asyncio.Task exits cleanly; `_orch_done` sees
     `exc is None` and is a no-op.
   - AB-09 Layer-2 zombie guard releases the `linear_project_id` slot
     because the orchestrator task transitions to done; the next
     kickoff for the same project succeeds without a daemon restart.

## Architectural notes

### API surface decision (deviation from prompt)

The original prompt suggested `PATCH /v1/mesh/tasks/{id}/cancel` or
`PATCH /v1/mesh/tasks/{id}` with `status: "canceled"`. soul-svc v2.0.0
exposes neither: `/openapi.json` confirms only `PATCH .../claim` and
`PATCH .../complete` exist on a mesh task, and the `TaskCompleteRequest`
status enum is `completed | failed`. **No new soul-svc endpoint is
added**; instead, the orchestrator polls its own kickoff task via the
existing `GET /v1/mesh/tasks?status=...` listings and recognises three
cancel signal shapes (matches the SAL-2756 design sketch verbatim:
"reuses existing /complete PATCH semantics"). When soul-svc later adds
a `canceled` status, no orchestrator change is needed — the
case-insensitive match already handles it.

### Polling interval choice

`_check_cancel_signal` is called once at the top of each
`_dispatch_wave` tick (every `poll_sleep_sec`, default 45s). Reasons:

- already on the dispatch loop's hot path → no separate timer.
- one extra `list_tasks(status="failed")` round trip per tick is cheap
  (already polled by `_poll_children` for completed children; could be
  collapsed in a future refactor but kept separate here for diff
  focus).
- 45s observation latency is well under the 60s "daemon restart" cost
  the ticket targets eliminating.

### Where the cancel check sits in the loop

Top of the tick, BEFORE `_select_ready` / dispatch loop. This guarantees
the cancel-tick produces zero new dispatches even if there is ready
work — `_drain_mode` is set in the same tick.

### Drain semantics vs `task.cancel()`

We do NOT call `asyncio.Task.cancel()` on either the orchestrator
itself or its in-flight children. Reasons:

- Children are out-of-process (mesh-dispatched subs). The local Task
  object is just the orchestrator's own polling loop; cancelling it
  mid-poll would skip checkpointing.
- Letting in-flight children finish matches the user prompt's "let
  current children finish, don't dispatch new ones, exit cleanly"
  contract and preserves SS-08 / Linear / soul-memory consistency.
- The graceful-exit path already routes through the standard
  `_orch_done` callback (no exception → no-op). AB-17-n's belt-and-
  suspenders for unexpected crashes still covers the case where the
  cancel-aware path itself throws.

## Risks

- **Stuck child task during drain.** If a dispatched child never
  completes (zombie sub), the wave's `still_in_flight` check stays
  non-empty and the cancel never breaks the loop. Mitigation: existing
  AB-17-n / AB-17-p deadlock + stall detectors still fire and will
  coerce non-terminal tickets to FAILED; once they do, the cancel
  early-exit fires on the next tick. Worst case is the same as the
  pre-AB-17-q stall window, which AB-17-p / AB-17-n already bound.
- **Race with operator double-PATCH.** If the operator PATCHes
  `complete(status="failed")` twice (e.g. retries on perceived hang),
  `_complete_kickoff_canceled`'s catch-all `except` swallows soul-svc's
  409 / "already terminal" response. Idempotent.
- **Cancel during `_run_on_all_green_actions`.** Not handled in this
  PR — once all waves finish green, on_all_green is allowed to run to
  completion. Future work if it becomes a problem; the on_all_green
  block is short-lived (single-PR merges, slack post) compared to the
  multi-hour wave loop the cancel targets.
