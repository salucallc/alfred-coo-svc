# SAL-2617: Endpoint memory push loop

## Target paths
- src/alfred_coo/fleet_endpoint/__init__.py
- src/alfred_coo/fleet_endpoint/memory_push.py
- tests/test_fleet_memory_push.py
- plans/v1-ga/SAL-2617.md

## Acceptance criteria
> 1000 writes reflected in hub `fleet_memory_sync_log` within 60s; `local_seq` monotonic; dup push idempotent

## Verification approach
- Unit tests in `tests/test_fleet_memory_push.py` simulate 1000 memory push calls to the `/v1/fleet/memory/push` endpoint and assert that the hub's `fleet_memory_sync_log` contains 1000 entries within a 60‑second window.
- Integration test spins up a temporary hub and endpoint, runs the memory push loop with a 30‑second cadence, then queries the log to ensure monotonic `local_seq` and idempotent handling of duplicate pushes.
- CI pipeline includes `ruff` linting and `pytest` execution; both must pass.

## Risks
- Potential performance impact if push batch size is large; mitigated by limiting batch to 50 items per request.
- Idempotency relies on correct `local_seq` handling; ensure strict increment and duplicate detection logic.
- Dependency on F06 (heartbeat) and F08 (soul-lite) being stable; failures in those components could mask issues in this feature.
