# SAL-2870 — Dependency-aware retry + deadlock grace + topo dispatch

Linear: https://linear.app/saluca/issue/SAL-2870
Branch: `feature/sal-2870-dispatch-retry-grace-topo`
Related: AB-17-n (immediate-coerce introduced), AB-17-x / SAL-2831 (phantom reconcile), AB-17-y / SAL-2842 (orphan-active reconcile)

## Why this exists

v7o crashed at 18:09:19 UTC on 2026-04-25 with:

```
wave 1 deadlock: 17 tickets non-terminal with no in-flight or ready; coercing to FAILED:
['SAL-2676', 'SAL-2675', 'SAL-2674', 'SAL-2673', 'SAL-2656', 'SAL-2650', 'SAL-2620',
 'SAL-2613', 'SAL-2612', 'SAL-2611', 'SAL-2610', 'SAL-2601', 'SAL-2588', 'SAL-2587',
 'SAL-2586', 'SAL-2585', 'SAL-2584']
[wave-gate] wave=1 green=1 failed=19 excused=8 denominator=20 threshold=0.50 ratio=0.05 decision=failed_below_threshold
```

The 17 coerced tickets weren't dispatched because their deps weren't MERGED yet. When upstream tickets (SS-10, others) failed, downstream BLOCKED forever. AB-17-n's same-tick coerce-to-FAILED cascaded the entire wave-1 tail to FAILED before any retry could land.

## Five components

### 1. Per-ticket retry budget

`Ticket.retry_budget: int = 2` and `Ticket.retry_count: int = 0`. On any FAILED transition, a post-pass at the end of `_poll_children` checks `retry_count < retry_budget`; if true, route to BACKED_OFF instead of terminal FAILED, increment `retry_count`, clear `child_task_id` so the next dispatch creates a fresh sub.

Default `retry_budget=2` is configurable per-kickoff via top-level `retry_budget` payload field. `retry_budget=0` restores legacy "FAILED is terminal" behaviour and is used by 4 existing tests that pin the terminal-FAILED transition surface.

### 2. BACKED_OFF state with auto-recovery

New `TicketStatus.BACKED_OFF = "backed_off"` (NOT in `TERMINAL_STATES`). New `Ticket.backed_off_at: float | None` timestamp. `_wake_backed_off_tickets()` runs at the top of `_poll_children` and flips BACKED_OFF → PENDING when `now - backed_off_at >= retry_backoff_sec`.

Default `retry_backoff_sec=300` (5 min), configurable via `retry_backoff_sec` payload field. Setting `0` collapses the cooling window for tests/dry-run.

### 3. Re-evaluate downstream readiness on every tick

`_refresh_blocked_status()` runs at the top of `_poll_children` and walks every ticket: BLOCKED tickets whose deps are now MERGED_GREEN flip to PENDING; PENDING tickets whose deps are not satisfied flip to BLOCKED. Idempotent + cheap (single graph pass, no I/O).

The previous unblock path was inside `_select_ready` which only triggered on dispatch consideration — a downstream that had been BLOCKED for many ticks while its upstream cycled FAILED → BACKED_OFF → MERGED_GREEN could miss the unblock event. The new pre-pass closes that gap.

### 4. Deadlock detector grace period

`_dispatch_wave`'s AB-17-n branch and `_wait_for_wave_gate`'s parity branch now arm a `_no_progress_since` timer when `in_flight=0 + ready=0` is observed. Coerce-to-FAILED only fires after `now - _no_progress_since >= deadlock_grace_sec`. Reset to `None` whenever in_flight or ready (or, in the gate branch, BACKED_OFF cooling) is non-empty.

Default `deadlock_grace_sec=900` (15 min), configurable via `deadlock_grace_sec` payload field. `_no_progress_since` is persisted across snapshots so a daemon bounce mid-grace-window resumes the same start point rather than restarting the timer.

The `wait_for_wave_gate` branch additionally treats BACKED_OFF as forward-progress so a wave with cooling tickets but no in-flight doesn't trigger the gate's narrower coerce path.

### 5. Topological dispatch ordering within wave

`_topo_sort()` is a Kahn-style sort over a candidate ticket slice using `blocks_in` / `blocks_out` edges restricted to the slice. Cycles fall through with the unreachable remainder appended in identifier order so the function always returns `len(input)` items (with a WARN log).

`_select_ready` now applies (CP-tier sort) → (identifier-sort) → (per-CP-tier topo sort) so the final order respects: critical-path first, then deps before dependents within each tier, then identifier as the tiebreaker.

## Architectural decisions

1. **Sweep at end of `_poll_children` instead of wrapping every FAILED transition.**
   The orchestrator has 11+ sites that write `ticket.status = TicketStatus.FAILED`. Wrapping each one with retry logic would have an unbounded conflict surface with the parallel SAL-2869 sub. The end-of-tick sweep is surgical: every existing FAILED-transition codepath stays untouched and the retry verdict is applied uniformly afterwards.

2. **`BACKED_OFF` not in `TERMINAL_STATES`.**
   This is required for the wave-gate to keep waiting through the cooling window. The cost is that the deadlock detector now needs the grace period (component 4) to coexist with non-zero cooling — they're tuned together.

3. **`retry_budget` persisted per-ticket, default kickoff-configurable.**
   `_apply_restored_status` seeds the kickoff default onto every ticket FIRST, then per-ticket restores override on top. This means a kickoff that bumps `retry_budget=5` mid-run gets the new default applied to fresh tickets while preserving any in-progress retry counts.

4. **Topo sort runs within CP tier, not across.**
   Critical-path always wins because the operator may have flagged a critical-path-leaf-but-deep-tree ticket as the priority pin (e.g. SS-08 for a post-gap rerun). Topo-within-tier preserves CP semantics while eliminating the random ordering of equal-priority tickets.

5. **`_no_progress_since` persisted across snapshots.**
   A daemon bounce inside the 15-min grace window MUST resume the same timer rather than restart it; otherwise an operator restart-as-cancel loop could indefinitely defer the deadlock coerce. Persistence ensures the timer is forward-only.
