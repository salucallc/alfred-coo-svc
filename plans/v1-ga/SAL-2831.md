# SAL-2831: [AB-17-x] Reconcile in_flight set against mesh state (post-v7k unstick)

Linear: https://linear.app/saluca/issue/SAL-2831

## Bug — observed twice on 2026-04-25

**v7i** (kickoff `79903320-9477-484e-9c47-3b18496c10c6`, started 06:11 UTC).
Wave 0 cleared; wave 1 dispatched 11/28 tickets. Last forward progress
~06:42 UTC. First `[watchdog] no forward progress` at 06:57 UTC. Stuck
forever at `[cadence] wave=1 tickets=1/28 in_flight=1 spend=$1.80/$30.00`.

**v7k** (kickoff `a1049c5f-1155-4318-b735-3c70ee7e613e`, started 07:14 UTC).
Same pattern: 11/28 dispatched, wedged at the same `tickets=1/28
in_flight=1` count. First watchdog at 07:55 UTC. Drained 110+ minutes
later by an external cancel from a duplicate-kickoff rejection at 07:46
UTC, but the cancel-exit at `orchestrator.py:1638` requires
`still_in_flight` to be empty — and `still_in_flight` stayed at `[SAL-2672]`
forever, so the drain never completed.

Both runs hung on the **same ticket**: SAL-2672 SS-11 fix-round-1.
Mesh task `c18a2c2e-...` (v7k) / `29281a6a-...` (v7i) both completed
within ~20s of claim, with `iterations: 4` and `summary: "Opened a
Linear issue to resolve missing target information for SAL-2672"`. The
model went off-script on the respawn — instead of calling `propose_pr`
or `update_pr` against the existing PR (per the AB-17-o respawn body),
it created a Linear "grounding gap" issue and exited cleanly. The
mesh task is `status=completed` but its `result` envelope contains no
PR URL anywhere (verified: `_extract_pr_url(result)` returns `None`
for both records).

The orchestrator's in-flight set, however, kept SAL-2672 in
`DISPATCHED` / `IN_PROGRESS` for hours despite the mesh-side ground
truth showing zero claimed tasks for the run. The watchdog
(AB-17-p) observed the stall but is visibility-only: it WARN-logs
every 45s without taking action, by design.

## Root-cause hypothesis (defensive scope)

The exact path that prevents `_poll_children` from transitioning
SAL-2672 into FAILED on the post-`c18a2c2e`-completion tick was not
isolated from logs alone (the relevant tick boundary lacked the
ticket-state-transition events that would pin it down). The patch
treats this as a class-of-bug rather than a single point fix:

- **Class A (mesh-state lag).** A brief window between the executor's
  `PATCH /complete` and the next `?status=completed` query CAN return a
  list that does not yet contain the just-completed task. soul-svc
  v2.0.0 has no documented read-after-write SLA on the listing route.
  Sub-second in practice but unbounded under load.

- **Class B (single lifecycle blind spot).** `_poll_children` only
  fetched `status=completed`. Mesh tasks that the executor marked
  `status=failed` (main.py:585, on dispatch crash) were invisible —
  the ticket would sit DISPATCHED until the wave deadlock detector
  caught a fully-stuck graph. AB-17-q's cancel handler did mark the
  kickoff itself failed but child mesh tasks aren't.

- **Class C (off-script model output).** The respawned alfred-coo-a
  child for SAL-2672 SS-11 returned a clean structured envelope but
  with no `propose_pr` / `update_pr` tool call. The existing FAILED
  branch at orchestrator.py:2059 should have caught this; that it
  didn't (across both v7i + v7k) suggests Class A or another race we
  cannot pin from log evidence alone.

The patch addresses all three with a single reconciliation pass: if a
ticket's `child_task_id` is in NONE of the three mesh lifecycle states
(`claimed`, `completed`, `failed`) AND the ticket has been
DISPATCHED/IN_PROGRESS for more than `STUCK_CHILD_FORCE_FAIL_SEC`
(30 min default), the orchestrator force-fails the ticket.

## Target paths

- `src/alfred_coo/autonomous_build/orchestrator.py`
  - new module constant `STUCK_CHILD_FORCE_FAIL_SEC = 30 * 60`
  - `_poll_children` rewritten to fetch `claimed` + `failed` in
    addition to `completed`, merge `completed` + `failed` into
    `terminal_by_id`, surface `claimed_ids`, and add a
    phantom-detection branch that force-fails after the threshold
- `tests/test_autonomous_build_orchestrator.py` — 4 new tests:
  - `test_poll_children_force_fails_phantom_after_threshold`
  - `test_poll_children_does_not_force_fail_below_threshold`
  - `test_poll_children_handles_mesh_failed_status`
  - `test_poll_children_keeps_claimed_child_in_progress`

## Acceptance criteria

- The orchestrator never gets stuck on a phantom `in_flight=N` count
  when mesh-state shows zero claimed tasks for the run.
- `_poll_children` reconciles internal in-flight set against mesh
  ground truth on every tick: `claimed | completed | failed` ∪ →
  state transition; absent from all three for > threshold → force
  FAILED.
- A child task that the executor marked `status=failed` is observed
  by `_poll_children` (was previously invisible) and rolls the
  ticket → FAILED with the mesh's `result.error` as the reason.
- Existing healthy-path behaviour is preserved: completed-with-PR →
  PR_OPEN → REVIEWING; completed-without-PR → FAILED (no change);
  REVIEWING / MERGE_REQUESTED tickets skipped (AB-08 dedup).

## Verification approach

1. **Unit tests** drive `_poll_children` against `_FakeMesh` records
   with deterministic `status` values:
   - phantom child + ancient `_ticket_transition_ts` → asserts
     `ticket.status == FAILED`, Linear → Backlog, event recorded
     with `note` containing `"phantom_child"`.
   - phantom child + recent transition → asserts ticket stays
     IN_PROGRESS (no false-fail under the brief mesh-lag window).
   - mesh-`failed` child → asserts ticket FAILED, Linear → Canceled,
     reason from `result.error`.
   - mesh-`claimed` child + ancient transition → asserts ticket
     stays IN_PROGRESS (in-flight is sacred, never force-fail a
     long-running child).
2. **Regression check.** All existing 100+ orchestrator tests
   continue to pass (new branches are additive; the
   completed-with-PR / completed-without-PR / failed-mesh-record
   paths are unchanged).
3. **Live evidence (post-merge).** Once deployed, an orchestrator
   tick that observes a non-claimed / non-completed / non-failed
   `child_task_id` will emit a `WARNING: AB-17-x: phantom child
   <id> for <ticket> ...` log line and a `ticket_failed` state
   event with `note=phantom_child:...`. v7i + v7k would have
   force-failed SAL-2672 30 min after the respawn, allowing the
   wave-gate / dispatch loop to advance.

## Caveats

- The 30-min threshold is conservative. A future patch could shrink
  it once we have confidence the heuristic is correct (no
  false-fails in a week of live runs). Cf. AB-05's
  `DEFAULT_STALL_THRESHOLD_SEC` which uses the same value for the
  CP stall-watcher Slack ping.
- `_ticket_transition_ts` is populated by
  `_snapshot_graph_into_state` which runs at the END of every
  dispatch tick. The first tick after a fresh respawn will see
  `entered_ts == None` → `stuck_for == 0.0` → no force-fail. This
  is intentional: respawn is a state transition; the next tick's
  snapshot will set `entered_ts` to ~that respawn time.
- The patch deliberately does NOT alter `_extract_pr_url` semantics
  or the dispatch-loop break conditions. Once SAL-2672 transitions
  to FAILED, the existing AB-17-n deadlock detector (line 1664)
  + cancel-drain exit (line 1638) handle the rest.
- Does not address the underlying model behaviour (alfred-coo-a
  going off-script and creating a Linear issue instead of pushing
  fixes). That's tracked separately as a model-prompt /
  iteration-cap tuning concern.
