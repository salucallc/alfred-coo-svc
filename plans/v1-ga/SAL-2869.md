auto-merge path breaks main.

## Fix scope

Three layers of defense, one shared helper.

### Layer 1 (preventive, prompt-only)

Append a `DELETION GUARDRAIL` clause to the `alfred-coo-a` system
prompt in `persona.py`. Builder is told the per-file ratio
(`min(0.7 * original_file_LOC, 500 LOC)`), the keyword override
(`rewrite`/`replace`/`nuke`/`reset`), and the per-PR ratio cap
(`2x` net deletions, `100`-line floor, `refactor` label override).

Hawkman's persona prompt also gains a `GATE 2.5` clause naming the
destructive-PR guardrail as a hard reject reason — so a careful
reviewer catches it before the orchestrator's programmatic override
has to fire.

### Layer 2 (verdict gate, programmatic) — **Option A chosen**

`AutonomousBuildOrchestrator._handle_review_verdict` runs
`_check_destructive_guardrail_for_ticket` BEFORE acting on an
APPROVE verdict. If the guardrail trips, the verdict is OVERRIDDEN
to `REQUEST_CHANGES` regardless of what hawkman emitted, the
override reason is appended to the respawn body via
`rec.result.summary`, and a `verdict_overridden_destructive` event
is recorded.

Why Option A over Option B (prompt-only): Option A is deterministic
and survives prompt drift / model swap. Option B is reinforcement
that runs in the same prompt-eval loop hawkman is already missing,
so it adds belt without much braces. We did Layer 1's hawkman
prompt edit ANYWAY for visibility, but the load-bearing path is
the programmatic override.

### Layer 3 (pre-merge static check)

`AutonomousBuildOrchestrator._merge_pr` runs the SAME helper one
more time as the last gate before invoking `github_merge_pr`. If
it trips at this layer (manual operator merge, restart-resume race,
future code path that bypasses Layer 2), the merge is REFUSED:

- `[merge-block]` ERROR log with citations
- `merge_blocked_destructive` state event
- Linear ticket transitioned to `Backlog`
- `linear_add_comment` posts the layer + citations on the issue
- `_merge_pr` returns False; caller (`_handle_review_verdict`)
  marks the ticket FAILED

## Shared helper

`src/alfred_coo/autonomous_build/destructive_guardrail.py` exposes
`compute_destructive_guardrails(pr_files, *, hint_description,
has_refactor_label, base_repo, base_ref, original_loc_lookup)` which
runs both gates against a `GET /repos/.../pulls/{N}/files`
response. First-trip wins (per-file checked first).

## APE/V citations

- **A** (assumption): The 7 destructive PRs from 2026-04-25 (PR #84,
  #27, #21, #65, #68, #69, #20) all share two structural traits: a
  single-file deletion exceeding `min(0.7 * original_LOC, 500)` OR
  a per-PR `deletions/additions` ratio above 2x with deletions
  > 100 LOC.
- **P** (procedure): Build a PR-files fixture for each canonical
  pattern (PR #84 = per-file gate; PR #27 = per-PR gate; PR #25 +
  #66 = clean refactors). Run `compute_destructive_guardrails`
  against each. Wire the helper into Layer 2 + 3 paths.
- **E** (evidence): All 16 unit + integration tests pass:
  - PR #84 fixture trips per-file gate (citation includes
    `deploy/appliance/docker-compose.yml`).
  - PR #27 fixture (spread to 22×100 to isolate from per-file)
    trips per-PR gate (citation `2200` deletions, 0 additions
    summed × 22 = 44 — wait, 22×2=44 — ratio 50x).
  - PR #25 (`+91/-24`) does NOT trip.
  - PR #66 (`+60/-58`) does NOT trip.
  - Refactor-labelled `+500/-1000` does NOT trip.
  - Per-file gate clamps at `500` for files >715 LOC original.
- **V** (verdict): Confirmed three-layer guardrail catches every
  destructive PR pattern observed 2026-04-25. Layer 2 overrides
  blind APPROVEs; Layer 3 catches anything that bypasses Layer 2.

## Constraint check

- Don't merge — PR opened, no merge.
- Threshold values hardcoded as first-pass; future ticket can make
  configurable.
- Keywords case-insensitive (`rewrite`/`replace`/`nuke`/`reset`).
- Refactor-label detection case-insensitive (`refactor`).
- All new wiring is best-effort: any infra failure (PR-files
  fetch, LOC lookup) returns a non-tripped result so we never
  block a legitimate merge on flaky GitHub API.
